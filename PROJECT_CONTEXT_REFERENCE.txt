=======================================================================
PROJECT OVERVIEW
=======================================================================

Repository Name: wp-theme-toolkit
Description: A toolkit of prompts, guides, examples, and helper scripts for
             AI-assisted WordPress site building with Blocksy, Meta Box AIO,
             and MB Views.

Purpose: Provide structured workflow documents for project bootstrap,
         session bootstrap, design-token generation, new page and CPT
         creation, view review, and launch-readiness QA.

Target Audience: WordPress developers building sites with the Meta Views
                 Stack in VS Code, Cursor, Windsurf, Copilot, or Codex-like
                 environments.

=======================================================================
CURRENT SHAPE
=======================================================================

- Repository type: Toolkit / documentation-first repo
- Dominant file types: Markdown, text references, helper scripts
- Runtime code surface: Minimal; operational logic mainly lives in
  d2-scripts/restore-point.ps1 and d2-scripts/restore-point.sh
- Target domain: Blocksy child-theme site projects using Meta Box AIO and
  MB Views, not generic theme-authoring or plugin internals

Core workflow order:
1. PROJECT_BOOTSTRAP_PROMPT.md for new projects
2. SESSION_BOOTSTRAP_PROMPT.md for new chats
3. RESTORE_POINT_PROMPT.md before edit-heavy work
4. Build prompts in ds2-build/
5. Review prompts in ds3-review/
6. Pre-launch prompts 01-06 in ds4-pre-launch/

=======================================================================
WORKING ASSUMPTIONS
=======================================================================

- The toolkit itself is not normally the target of its own prompts.
- The preferred target is a site repository or Blocksy child-theme root that
  contains style.css, functions.php, _project-context.md, mb-json/, or views/.
- Local copies of MB View Twig and CSS should be kept in the target repo even
  when the live rendering is pasted into the WordPress admin.
- Template assignments that live in the database must also be recorded in a
  placement map inside _project-context.md.

=======================================================================
DOCUMENTATION RISKS TO WATCH
=======================================================================

- The toolkit depends on consistent path references across README, quick
  reference, task runner, and prompts.
- The design system guide is the highest-leverage file; weak rules there will
  propagate directly into generated views.
- Database-resident MB Views and Blocksy assignments are easy to lose if the
  placement map is not kept current.
