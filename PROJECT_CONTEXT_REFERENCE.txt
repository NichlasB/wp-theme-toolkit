PROJECT OVERVIEW
=======================================================================

Repository Name: wp-theme-toolkit
Description: A documentation-first toolkit of prompts, guides, examples,
             checklists, and helper scripts for AI-assisted WordPress site
             building with Blocksy, Meta Box AIO, MB Views, and GridPane
             deployment.

Purpose: Provide structured workflow documents for start-here routing,
         project bootstrap, project status snapshots, Claude Design and visual
         handoff conversion, session bootstrap, session handoff, restore
         points, design-token generation, new page and CPT creation, view
         review, launch-readiness QA, deployment, LocalWP reverse refresh, Git
         operations, post-launch guidance, and toolkit retrospective learning.

Target Audience: WordPress developers building sites with the Meta Views
                 Stack in VS Code, Cursor, Windsurf, Copilot, or Codex-like
                 environments.

=======================================================================
CURRENT SHAPE
=======================================================================

- Repository type: Toolkit / documentation-first repo
- Dominant file types: Markdown, text references, helper scripts
- Runtime code surface: Minimal; operational logic mainly lives in
  d2-scripts/restore-point.ps1 and d2-scripts/restore-point.sh
- Target domain: Blocksy child-theme site projects using Meta Box AIO,
  MB Views, and Git-backed child-theme code deployment
- Master routing: START_HERE_MASTER_WORKFLOW.md is the root entry point when
  the next workflow is not obvious
- Long-break recovery: target projects can keep mvs-project-status.md beside
  _project-context.md
- Visual handoff support: Claude Design, screenshots, mockups, HTML exports,
  and existing-site captures are upstream design inputs, not production
  builder output
- Deployment coverage: documented GridPane deployment workflow plus a
  per-project deployment checklist and a LocalWP reverse refresh workflow
- Shared workflow sourcing: SESSION_BOOTSTRAP_PROMPT.md is a local adapter
  around the shared bootstrap core in sibling repo wp-workflow-toolkit/

Core workflow order:
1. START_HERE_MASTER_WORKFLOW.md when the next workflow is not obvious
2. PROJECT_BOOTSTRAP_PROMPT.md for new projects
3. DESIGN_HANDOFF_TO_MVS_PROMPT.md for Claude Design, mockup, screenshot, or
   HTML handoffs
4. SESSION_BOOTSTRAP_PROMPT.md for new chats or stale project resumes
5. RESTORE_POINT_PROMPT.md before edit-heavy work
6. Build prompts in ds2-build/
7. Review prompts in ds3-review/
8. Pre-launch prompts 01-06 in ds4-pre-launch/
9. GRIDPANE_DEPLOYMENT_PROMPT.md in ds5-deploy/
10. LOCALWP_REVERSE_REFRESH_PROMPT.md when local needs current GridPane
    content or missing uploads
11. GIT_OPERATIONS_PROMPT.md in ds6-git/

Optional toolkit-retrospective workflow:
- TOOLKIT_LESSONS_AUDIT_PROMPT.md in ds7-maintenance/ when a working chat
  exposes toolkit-wide lessons worth folding back into the toolkit itself

=======================================================================
HIGH-VALUE FILES
=======================================================================

START_HERE_MASTER_WORKFLOW.md
  - Root front-door workflow for new, stale, visual handoff, build, QA,
    deployment, and toolkit-improvement sessions
  - Classifies the session and routes to the next specialized prompt

_TASK_RUNNER.md
  - Entry point for prompt execution rules, target selection, task syntax,
    checklist expectations, and prompt-family scope

PROJECT_STATUS_TEMPLATE.md
  - Canonical schema for target-project mvs-project-status.md snapshots
  - Tracks current phase, last completed workflow, next recommended workflow,
    design source, editability level, blockers, and deployment state

PROJECT_CONTEXT_TEMPLATE.md
  - Canonical schema for each target project's _project-context.md
  - Includes design source, design approval, client editability, and editable
    surface map fields

CLAUDE_DESIGN_HANDOFF_WORKFLOW.md
  - Defines Claude Design and visual mockups as upstream design inputs, not the
    production WordPress builder

DESIGN_HANDOFF_TO_MVS_PROMPT.md
  - Converts Claude Design, screenshots, mockups, HTML exports, or existing
    site captures into MVS implementation decisions and routes to the next
    build prompt

SESSION_BOOTSTRAP_PROMPT.md
  - Local site-project adapter around the shared bootstrap core in
    wp-workflow-toolkit/
  - Reads project status when available and flags missing/stale status as a
    long-break recovery risk

STACK_REFERENCE.md
  - Single source of truth for the Meta Views Stack workflow, child-theme
    scaffold, required artifacts, and local reference-copy rules

DESIGN_SYSTEM_GUIDE.md
  - Authoritative design-token, layout, naming, responsive, and multi-section
    page guidance

WORKFLOW_QUICK_REFERENCE.md
  - Fast route map for operating the toolkit day to day

=======================================================================
WORKING ASSUMPTIONS
=======================================================================

- The toolkit itself is not normally the target of its own prompts, except
  when GIT_OPERATIONS_PROMPT.md or TOOLKIT_LESSONS_AUDIT_PROMPT.md is used
  intentionally against this repo.
- The preferred target is a site repository or Blocksy child-theme root that
  contains style.css, functions.php, _project-context.md, mb-json/, or views/.
- Target projects should maintain mvs-project-status.md beside
  _project-context.md when active tracking or long-break recovery matters.
- Local copies of MB View Twig and CSS should be kept in the target repo even
  when the live rendering is pasted into the WordPress admin.
- Template assignments that live in the database must also be recorded in a
  placement map inside _project-context.md.
- Claude Design and visual handoffs are planning inputs only; production output
  remains Blocksy child theme, Meta Box AIO, MB Views, and intentionally scoped
  Gutenberg only when the editability level calls for it.
- Code moves through Git; content and settings move through a database
  migration workflow.
- TOOLKIT_LESSONS_AUDIT_PROMPT.md is intentionally excluded from
  PRE_LAUNCH_CHECKLIST.md and DEPLOYMENT_CHECKLIST.md because it targets the
  toolkit itself rather than a site project.

=======================================================================
DOCUMENTATION RISKS TO WATCH
=======================================================================

- The toolkit depends on consistent path references across README, task
  runner, quick reference, prompt files, and both checklists.
- The master workflow should stay a router, not duplicate every specialized
  prompt.
- The status snapshot should stay concise and not replace _project-context.md.
- Visual handoff guidance must stay clear that exported HTML is not pasted
  directly into WordPress as the production implementation.
- Database-resident MB Views, Blocksy assignments, and HTML widget content are
  easy to lose if placement decisions and deployment notes are not recorded.
